1 · What is this tool?

The assessment is a single .html file. It contains everything — the questions, a real SQL database, a SQL engine, an editor, and an auto-grader. The candidate opens it in any modern browser. No internet connection, no logins, no software to install.

When the candidate finishes, the tool downloads a results file to their computer. They send that file to you. You open it in your browser to review their answers.

The file you send to candidates is the file in the dist/ folder ending in .html — for example, banking-incentive-campaigns.html. This is the only file you need to share.

2 · Before the session

24 HOURS BEFORE
  • Confirm the video call link (Zoom, Google Meet, Teams — any works) and share it with the candidate.
  • Send the candidate the Candidate Guide (candidate-guide.html). Ask them to read it beforehand so they are not learning the interface during the test.
5 MINUTES BEFORE
  • Send the candidate the assessment file (.html) — for example, banking-incentive-campaigns.html. Do not share this file any earlier.
  • Join the video call and ask the candidate to share their screen.
  • Verify you can see their browser clearly — you will be watching their screen throughout.
  • Ask them to have the assessment file ready to open but not yet opened. The timer starts the moment they click "Begin Test".
  • Let them know how much time they have (the tool has no countdown — you manage time).
The file must be opened fresh. If the candidate already opened it and clicked "Begin Test", the session is in progress and cannot be restarted. Have them close and reopen the file only if they have not yet clicked Begin.

3 · During the session

What you will see on their screen

The interface has three panels. On the left you can see the timer counting up and coloured dots next to each question number. On the center are the questions they are working on. On the right is a scratch-pad playground.

Status dots — at a glance

Dot colourMeaning
Grey (no dot)Not yet attempted
PASSCorrect answer submitted
FAILRan a query; answer is wrong
ERRORSQL has a syntax or runtime error
NOT PERMITTEDTried to run a blocked SQL statement

Your role during the test

This is primarily an observation session. Stay on the call, watch their screen, and take your own notes.

You can:
  • Clarify the business question if the candidate is confused about what is being asked (not how to write the SQL).
  • Confirm the expected data format if there is genuine ambiguity in the question wording.
  • Tell them how much time remains.
  • Ask them to talk through their thinking — "can you explain what you're trying to do?" — if you want to assess reasoning, not just the final query.
Do not:
  • Tell them which SQL functions or clauses to use.
  • Confirm or deny whether their approach is correct before they run it.
  • Hint at the schema or data shape beyond what is visible in the tool.
Note what you observe beyond the score: How do they approach a problem? Do they explore the data first? How do they debug when something fails? These observations matter as much as PASS/FAIL counts.

4 · Ending the assessment

You control the clock. The tool has no countdown timer and will not end automatically. When time is up (or the candidate signals they are finished):

  1. Tell the candidate: "Please click the Finish Work button in the sidebar whenever you're ready."
  2. A confirmation dialog appears on their screen — they confirm.
  3. The screen changes to "SESSION LOCKED" and a results file downloads automatically.
  4. A short code appears on the locked screen under the heading "Report Integrity Fingerprint" — for example, A3F2-B891-7C4D-E520. Screenshot or write down this code immediately. You will use it to verify the report file has not been altered before you receive it.
Once they click Finish Work, the session is locked permanently. They cannot re-open the test in the same browser. Do not ask them to click Finish until you are sure the session is complete.

5 · Getting results back

When the candidate clicks Finish Work, a file named something like Banking_Incentive_Campaigns_results.html downloads automatically to their default Downloads folder.

How to receive the file

Ask the candidate to send you this file. Common options:

  • Email it as an attachment.
  • Upload to a shared folder (Google Drive, Dropbox, etc.).
  • If you are still on the video call, they can share it via the chat feature of your video tool.

If the download didn't happen

If the candidate reports no file downloaded, check these in order:

  1. Check the Downloads folder — it may have downloaded silently without a notification.
  2. Check the browser's download bar — some browsers (Firefox, Edge) show downloads in a bar at the bottom or top of the window.
  3. Pop-up blocker active: A blue banner should appear on the locked screen saying "Your results have been saved as a downloaded HTML file." This confirms the download did complete — they just need to look in their Downloads folder.
  4. If there is truly no file, ask them to check for any browser download permission dialogs they may have dismissed.

6 · Verifying report integrity

The results file includes a tamper-evidence mechanism. Because the file lives on the candidate's machine between submission and delivery to you, this lets you confirm it has not been edited.

How it works

When the candidate clicks Finish Work, the tool computes a fingerprint from their answers, SQL, and notes. This fingerprint appears in two places:

  • On the locked screen (visible to you via screen share) — the "Report Integrity Fingerprint" code, e.g. A3F2-B891-7C4D-E520.
  • At the top of the results file — the same code, plus an automated VERIFIED / INTEGRITY CHECK FAILED banner computed when the report is opened.

Step-by-step verification

  1. During the call: When the locked screen appears, screenshot or write down the fingerprint code.
  2. When you receive the file: Open the _results.html file in a browser. The fingerprint is shown in a box at the very top of the page.
  3. Check the code matches what you recorded from the screen.
  4. Check the VERIFIED banner — the report runs an automatic integrity check when opened and displays a green ✓ VERIFIED message if nothing has been changed, or a red warning if it detects a discrepancy.
The automated check catches simple edits (changing FAIL to PASS in the HTML text). The fingerprint code you recorded during the live session is the stronger safeguard — a sophisticated edit would need to produce the exact same code, which is computationally infeasible.
The fingerprint was not recorded? If you forgot to note the code during the call, the automated VERIFIED banner in the report is still useful — it catches naive modifications. If the banner shows VERIFIED, the report is almost certainly unaltered.

7 · Reviewing the results

Open the _results.html file in any browser. No internet connection needed.

What the report contains

  • Candidate name and timestamps — when the test started, when it was exported, total elapsed time.
  • Overall score — number of questions passed out of the total.
  • Notes — anything the candidate typed in the notes area during the test.
  • Per-question breakdown — the question prompt, expected columns, the SQL they submitted, their final status (PASS / FAIL / ERROR / NOT ATTEMPTED), and the result rows they got back.
  • Playground queries — all successful exploratory queries from the scratch-pad, in the order they were run.

How to interpret the score

StatusWhat it means for your evaluation
PASS The candidate's final submitted SQL produced the exact correct result set. Consider this a solid signal.
FAIL The query ran but the result was wrong. Read the SQL they wrote — a near-miss on a hard question is very different from a completely wrong approach on an easy one.
ERROR A syntax or runtime error. This often means the candidate had the right idea but didn't finish debugging. Look at the SQL to judge intent.
NOT ATTEMPTED No SQL was run for this question. Could be a time management issue or they skipped it intentionally.
The playground queries tell the full story. A candidate who explored the data methodically before writing answers — even if they didn't all PASS — is demonstrating strong analytical process. Review the playground section carefully.

8 · Guidance you can give candidates

These are things you are permitted to tell a candidate if they ask during the test:

Candidate asks…You can say…
"What tables are in the database?" Point them to the Schema Reference panel on the right side of the tool — it lists all tables and columns.
"Can I see example data?" Yes — tell them to run SELECT * FROM table_name LIMIT 10 in the playground to inspect any table.
"My query is blocked / NOT PERMITTED" Only SELECT queries and CREATE TEMP TABLE are allowed. Tell them to rewrite as a SELECT.
"My answer looks right but it keeps failing" Tell them to check that their selected columns match the Expected Columns list in the right order, and that they are not selecting extra columns.
"Can I use CTEs / window functions?" Yes — all standard SQLite features are supported, including CTEs and window functions.
"Is there a time limit?" Tell them however much time remains in the session.

9 · Troubleshooting

The file won't open

  • Make sure they are double-clicking the file (not trying to open it from inside an email without saving first).
  • Any modern browser works: Chrome, Firefox, Safari, Edge. Internet Explorer is not supported.
  • On macOS, if a security dialog appears saying the file is from an "unidentified developer", they should right-click → Open to bypass it.

The screen shows "SESSION LOCKED" immediately on open

This means the candidate (or someone else on that browser) already completed this assessment file previously. The lock is stored in the browser. Options:

  • Have them open the file in a different browser (e.g. Chrome instead of Safari).
  • Have them open the file in a private/incognito window.
  • If neither is possible, contact whoever manages the assessment to get a fresh copy of the file.

SQL queries are not running — "Database not yet initialized"

The SQL engine loads when the test begins. If this error appears, wait a few seconds and try again. On slow machines this can take 5–10 seconds. If it persists, refreshing the page will clear the session and require them to start over — so only do that if the test has not yet started.

The results file was not downloaded

See Section 5. The download usually completes silently. Ask them to check their Downloads folder before concluding the file is missing.

Candidate can't find the Finish Work button

It is at the bottom of the left sidebar. If the sidebar is collapsed, they need to click the › arrow at the top-left to expand it.

The tool looks broken or unstyled

This can happen if the browser is in an unusual mode (e.g. Reader Mode). Ask the candidate to exit Reader Mode and reload the file.
