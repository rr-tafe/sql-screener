Module Builder Guide — SQL Assessment Framework

How to create a new SQL assessment module from scratch and produce a distributable candidate file.

Part A — Domain Expert

You know the subject matter. You will use an LLM (Claude, M365 Copilot, ChatGPT) to generate the module content. No coding required. Hand off the output to a developer to build (or do it yourself by following the steps in Part B).

Part B — Developer

You receive the generated module files and run the build pipeline to produce the final candidate .html file.

Overview

Building a module is a two-phase process:

  1. Content generation (Part A) — A domain expert uses an LLM prompt template to produce all the module files: schema, data, questions, and reference answers.
  2. Build & packaging (Part B) — A developer runs a Python script that validates the content, runs all reference answers against the database to verify them, and packages everything into a single self-contained .html file ready to send to candidates.

The two parts are intentionally decoupled. The domain expert does not need any technical tools. The developer does not need to know the subject matter — the build script validates the content automatically.

Part A — Domain Expert: Generating Module Content

Start here if you are designing the assessment questions and data.

A1 · Use the LLM template

1
Open the template file.

Find template/MODULE_TEMPLATE.md in the project. Open it in any text editor and copy the entire contents.

2
Paste into Claude or ChatGPT.

Start a new conversation, paste the template, and send it. The LLM will begin an interview to understand your module requirements.

3
Answer the interview questions.

The LLM will ask up to eight questions. Answers guide all the generated content, so be specific:

  • Domain / industry — e.g. "retail banking", "e-commerce logistics"
  • Target role and seniority — e.g. "mid-level Business Analyst"
  • Business scenario — the fictional context the candidate will work within
  • Question count — typically 4–8 questions
  • Difficulty distribution — e.g. "2 easy, 3 medium, 1 hard"
  • Data volume tier — small (dozens of rows), medium (hundreds), or large (thousands). Larger = more realistic, but larger file size.
  • SQL concepts to test — e.g. "JOINs, GROUP BY, window functions, date arithmetic"
  • Data quality edge cases — e.g. "NULL values in amounts, duplicate enrollments"
4
Wait for the generated output.

After the interview, the LLM will produce all seven module files in a single response. Do not interrupt it mid-generation.

If the LLM's response is cut off (this can happen with very large modules), ask it to "continue from where you left off." You may need to do this a couple of times for large data volumes.

A2 · Review the LLM output

Before doing anything else with the zip, open the files and spend 10–15 minutes reading them. Focus on content, not formatting — the build script validates structure automatically.

Open files as read-only where possible. Several files (questions.json, solutions.sql, module.json) have a strict format that the build script enforces exactly. Accidentally saving a file with a stray character or encoding change can cause a validation failure. View them in a text editor rather than a spreadsheet tool, and resist the urge to reformat anything.

Questions

  • Read each question prompt as a candidate would. Is the business intent clear?
  • Check the expected columns list for each question — do they match what the reference SQL in solutions.sql actually selects?
  • Make sure no question reveals the answer to another question.

Data

  • Spot-check a few CSV rows for plausibility — dates in the right format, numeric amounts in a sensible range, foreign key IDs that exist in the referenced table.
  • Confirm the LLM included the data quality edge cases you asked for (NULLs, duplicates, etc.).

Reference answers

  • You do not need to run the SQL — the build step does that. But read each solution in solutions.sql to confirm it is answering the right business question.
Do not add plaintext expected results to any file. The reference answers are hashed by the build script — if a candidate can read the expected values from the file, the assessment is compromised.

A3 · Place or pass the module

The LLM delivers a zip file named <module-id>.zip. Inside it is a single folder — <module-id>/ — already structured correctly. You do not need to rename or reorganise anything.

Choose the path that fits your situation:

A
You are also the developer (running the build yourself).
Extract the zip and move the extracted <module-id>/ folder directly into the modules/ directory of the project. Then proceed to Part B.
B
A separate developer will run the build.
Send them the zip file as-is — no need to extract it first. Include a brief note on what the module covers and the intended candidate profile (role and seniority level), so they have context if the build raises questions. The developer extracts the zip, drops the folder into modules/, and runs the build script.
If you need to make any corrections after reviewing (A2), edit the relevant file inside the extracted folder before handing it off. Do not edit files inside the zip directly — extract first, edit, then re-zip if you need to send it to someone.

A4 · What the developer does with it

The developer runs the build script against the module folder. If all reference answers pass validation, they produce a single .html file in the dist/ folder. They will send this back to you to review before distributing it to candidates. See Part B for the full developer workflow.

Part B — Developer: Building and Packaging

Start here once the module folder is ready. You will validate the content, run the reference solutions, and produce the distributable HTML file.

B1 · Prerequisites & one-time setup

Required

ToolMinimum versionCheck
Python3.8python3 --version
Node.js18node --version
npmanynpm --version

One-time npm install

Run this once per machine, from the sql-screening/ directory:

cd sql-screening/runtime
npm install

This installs CodeMirror, sql.js, esbuild, and jsdom. If node_modules already exists and you haven't changed package.json, you can skip this.

The builder has no Python dependencies beyond the standard library. You do not need to create a virtualenv or run pip install.

B2 · Run the builder

From the sql-screening/ directory, run:

python3 builder/builder.py --module=modules/<module-id>

For example:

python3 builder/builder.py --module=modules/retail-credit-scoring

What happens during the build

The builder runs 13 steps in sequence. You will see numbered progress output:

StepWhat it does
[1]Checks all required files exist and are valid JSON/SQL
[2]Confirms engine is "sqlite"
[3]Confirms questionCount matches the number of questions in questions.json
[4]Builds an in-memory SQLite database from the schema and CSV data
[5–6]Runs every reference solution and computes an answer hash for each
[7]Validation gate — every solution must match its hash. Fails the build if any solution errors or produces unexpected results
[8]Derives a schema map for the editor's autocomplete feature
[9]Serialises the SQLite database to base64 for embedding in the HTML
[10]Assembles the final HTML, inlining all assets
[11]Writes output files to dist/
[12]Runs the JavaScript test harness to verify the assembled HTML
[13]Reports file size; warns if over 5 MB
The build is all-or-nothing. If any step fails, no output file is written (and any previously written output is deleted). Fix the reported error and re-run.

Successful output

A successful build ends with:

Build complete: …/dist/retail-credit-scoring.html

Two files are written to dist/:

  • <id>.html — the candidate assessment file
  • <id>-README.md — the module readme (copied from the module folder)

B3 · Understand the output

The dist/<id>.html file is the only file candidates need. It is fully self-contained — the SQL engine, database, all questions, and the answer hashes are embedded inside it. You can email it, upload it to Google Drive, or put it on a shared network drive.

Before distributing: Open dist/<id>.html in your browser and complete a few questions yourself to verify the UX. The test harness verifies correctness, but a manual check catches presentation issues.

File size

The tool embeds sql.js (the SQL engine) as a WebAssembly binary. A typical build is 1.5–2 MB. This is normal. The builder will warn (but not fail) if the file exceeds 5 MB — this usually means the data CSVs are very large.

B4 · Common errors and fixes

Error messageWhat it means and how to fix it
Required file missing: …/questions.json A required module file is absent. Confirm the extracted zip folder was placed directly inside modules/ without renaming. Ensure filenames match exactly (case-sensitive on Linux/macOS).
Missing CSV for table 'X' The schema defines a table named X but there is no data/X.csv. Either add the missing CSV or remove the table from schema.sql.
questions.json is not valid JSON JSON syntax error. Paste the file into jsonlint.com to find the exact line. Common causes: trailing commas, unescaped quotes inside strings.
questionCount mismatch module.json says there are N questions but questions.json has a different count. Update the questionCount field in module.json.
No solution block found for 'q3' solutions.sql is missing the -- q3 delimiter for question 3. Check that every question has a corresponding -- qN comment and that they are in order.
Solution for 'q2' raised an exception: … The reference SQL for question 2 has a runtime error (bad column name, wrong table name, etc.). Fix the SQL in solutions.sql. The full SQLite error is included in the message.
Solution validation gate failed One or more solutions errored. Individual errors are listed above this message. Fix them and re-run.
sql.js WASM file not found … Run 'npm install' npm install has not been run in runtime/, or the packages were not installed correctly. Run cd runtime && npm install.
JS test failed: integration_test.js The assembled HTML failed the automated test harness. This is rare and usually indicates a mismatch between the solutions and the expected answer hashes. Review the test output above the error line.

Debugging a solution failure

If a reference solution fails validation, the fastest way to debug it is to:

  1. Run the schema and data in any SQLite tool (e.g. the DB Browser for SQLite) to create a test database.
  2. Paste the failing solution SQL and run it.
  3. Compare the result to what the question's expectedColumns specifies.
  4. Fix the solution in solutions.sql and re-run the builder.

B5 · Distributing the file

Once the build succeeds, share dist/<id>.html with the screener. The screener guide explains how to forward it to candidates.

Each candidate gets the same file

The assessment file is not personalised — you send the same .html to every candidate for a given module. Each candidate works in their own browser, so sessions are isolated on their own machine.

Running multiple modules

Each module is independent. Run the builder once per module. Each produces its own dist/<id>.html file. You can have many modules in modules/ and build them at any time.

Rebuilding after changes

If you edit any module file (questions, data, solutions), re-run builder.py. The output HTML is overwritten only if the new build passes all validation. There is no incremental build — the full pipeline runs each time.

Do not distribute a file from a previous build if you have made changes to the module content. Always rebuild and test before sending a new batch of candidates the updated file.
